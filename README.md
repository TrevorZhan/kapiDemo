# kapiDemo — Minimal iOS ProRAW Camera with LUT Processing

A single-screen iOS camera app (Swift + UIKit) that captures Apple ProRAW photos (or JPEG/HEIC on unsupported devices), applies a color-grading LUT, and saves the processed image to the photo library.

## Implementation Approach

### ProRAW Detection and Activation

ProRAW support is detected and enabled in `CameraManager.swift` during session configuration:

1. **Session setup** — An `AVCaptureSession` is created with the `.photo` preset (`AVCaptureSessionPresetPhoto`), a back camera input, and an `AVCapturePhotoOutput`.
2. **Detection** — After adding the photo output to the session, we query `photoOutput.isAppleProRAWSupported` (available iOS 14.3+). This property returns `true` only on iPhone 12 Pro and later Pro/Pro Max models.
3. **Activation** — If supported, we set `photoOutput.isAppleProRAWEnabled = true`, which tells the output to make RAW pixel formats available for capture. A boolean `isProRAWSupported` is stored and used to:
   - Update the UI label ("ProRAW mode" vs "Fallback: JPEG/HEIC")
   - Choose between RAW and default capture settings at photo time
4. **Capture settings** — When the user taps capture, if ProRAW is supported, we create `AVCapturePhotoSettings` with `rawPixelFormatType` from `photoOutput.availableRawPhotoPixelFormatTypes`. Otherwise, we use default `AVCapturePhotoSettings()` which produces JPEG/HEIC.

### Image Processing Pipeline

The full pipeline from shutter press to saved photo flows through four files:

```
[Capture Button Tap]
       |
       v
CameraManager.capturePhoto()
  - Creates AVCapturePhotoSettings (ProRAW or JPEG/HEIC)
  - Calls photoOutput.capturePhoto(with:delegate:)
       |
       v
CameraManager.photoOutput(_:didFinishProcessingPhoto:error:)  [delegate callback]
  - Extracts image data via photo.fileDataRepresentation()
  - Reads orientation from photo EXIF metadata (kCGImagePropertyOrientation)
  - Passes data + isRAW flag + orientation to ImageProcessor
       |
       v
ImageProcessor.processImage(data:isRAW:orientation:)
  |
  |-- Step 1: DECODE
  |     - ProRAW: CIRAWFilter(imageData:identifierHint:) → CIImage
  |     - JPEG/HEIC: CIImage(data:) → CIImage
  |
  |-- Step 2: APPLY LUT
  |     - Parse 典雅绿调.cube (33x33x33, DaVinci Resolve format)
  |       - Read LUT_3D_SIZE to get cube dimension
  |       - Parse each line of R G B floats, append alpha = 1.0
  |       - Pack into contiguous Data buffer (Float array → Data)
  |     - Create CIColorCubeWithColorSpace filter
  |       - Set inputCubeDimension = 33
  |       - Set inputCubeData = parsed float buffer
  |       - Set inputColorSpace = deviceRGB
  |       - Set inputImage = decoded CIImage
  |     - Output: color-graded CIImage
  |
  |-- Step 3: RENDER
        - CIContext.createCGImage() → CGImage (raw pixels, no orientation)
        - Wrap as UIImage(cgImage:scale:orientation:) with original EXIF orientation
        - Output: final UIImage ready to save
       |
       v
PhotoSaver.save(image:completion:)
  - Requests PHPhotoLibrary authorization (.addOnly)
  - Calls PHAssetChangeRequest.creationRequestForAsset(from:)
  - Saves processed UIImage to Photos library
```

**Important**: The LUT is only applied to captured photos, never to the live camera preview.

### File Structure

| File | Responsibility |
|------|---------------|
| `ViewController.swift` | UIKit UI (fullscreen preview, status label, capture button), delegates to CameraManager |
| `CameraManager.swift` | AVCaptureSession lifecycle, ProRAW detection, photo capture, delegate callback |
| `ImageProcessor.swift` | RAW decoding (CIRAWFilter), .cube file parsing, LUT application (CIColorCubeWithColorSpace), final render |
| `PhotoSaver.swift` | PHPhotoLibrary permission request and image save |
| `典雅绿调.cube` | 33x33x33 3D LUT file (generated by DaVinci Resolve) |

## Main Issues Encountered and Solutions

### 1. Simulator Has No Camera

**Problem**: Running on the iOS Simulator shows "Camera unavailable" because `AVCaptureDevice` returns `nil` — simulators have no camera hardware.

**Solution**: Must test on a physical iPhone connected via USB or wireless debugging. ProRAW specifically requires iPhone 12 Pro or newer (Pro/Pro Max models only).

### 2. Placeholder LUT File Produced Wrong Colors

**Problem**: Captured photos did not show the expected color grading. The bundled `典雅绿调.cube` was a simplified 8x8x8 placeholder (512 entries with crude interpolated values) instead of the real LUT.

**Solution**: Replaced with the actual 33x33x33 LUT file (`典雅绿调 709 Lut.cube`, 35,937 RGB entries generated by DaVinci Resolve) which contains the correct color science for the elegant green tone look.

### 3. JPEG/HEIC Photos Had Wrong Orientation

**Problem**: When testing on an iPhone 14 (no ProRAW), photos taken in portrait appeared rotated 90° in the photo library. The iPhone camera sensor is physically oriented in landscape — photos only appear upright because of an EXIF orientation tag embedded in the metadata. However, `CIContext.createCGImage()` produces a raw pixel buffer with no metadata, so the orientation tag was lost and every image defaulted to landscape.

**Solution**: Read the `kCGImagePropertyOrientation` value from the captured photo's EXIF metadata in the delegate callback, pass it through the processing pipeline, and apply it when constructing the final `UIImage` via `UIImage(cgImage:scale:orientation:)`. A `CGImagePropertyOrientation` → `UIImage.Orientation` mapping extension handles the conversion between the two orientation enums.

### 4. ProRAW Capture Saved Two Photos Per Shot

**Problem**: When testing on a ProRAW-capable device, each capture saved two photos to the album — one with correct orientation and one without.

**Cause**: The capture settings included both `rawPixelFormatType` and `processedFormat: [AVVideoCodecKey: AVVideoCodecType.hevc]`. This tells AVFoundation to deliver two photos — one RAW and one HEVC. The delegate callback `didFinishProcessingPhoto` fired twice, and both were being processed and saved independently.

**Solution**: Keep the `processedFormat` parameter (required for RAW capture to function correctly), but check `photo.isRawPhoto` in the delegate callback. Only process and save the RAW photo; skip the processed companion with an early `return`.

### 5. ProRAW Photo Still Had Wrong Orientation After Fixing Duplicate

**Problem**: After fixing the duplicate save, the single remaining ProRAW photo was still rotated incorrectly. Debug logging revealed the cause: the EXIF metadata reported orientation **6** (rotate 90° clockwise), but `CIRAWFilter` had already applied that rotation internally during decoding — the output `CIImage` was already 3024×4032 (portrait pixels) with its own orientation property set to **1** (`.up`). By applying the EXIF orientation on top of already-rotated pixels, the image was being double-rotated.

**Solution**: For RAW photos decoded via `CIRAWFilter`, ignore the EXIF orientation from `photo.metadata` and instead read the orientation from the decoded `CIImage.properties` dictionary. Since `CIRAWFilter` already handles pixel rotation, this value is `.up`, and the final image renders correctly without any additional rotation.

### 6. SwiftUI Template Entry Point Conflict

**Problem**: The Xcode project was created with the SwiftUI App template (`@main struct kapiDemoApp: App`), which conflicted with our UIKit-based `ViewController` approach. The app showed "Hello World" instead of the camera.

**Solution**: Replaced the SwiftUI `@main` App struct with a UIKit `@main class AppDelegate: UIResponder, UIApplicationDelegate` that sets up a `UIWindow` with our `ViewController` as root, and configured the scene delegate for UIKit lifecycle.

## Requirements

- **Device**: iPhone (physical device required; simulator has no camera)
- **ProRAW**: iPhone 12 Pro or newer (Pro/Pro Max) for ProRAW mode; all other iPhones fall back to JPEG/HEIC
- **iOS**: 15.0+ (for `CIRAWFilter`; the app compiles for 14.3+ with fallback paths)
- **Permissions**: Camera access, Photo Library (add-only) access
